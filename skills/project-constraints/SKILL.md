# multiDemo 项目约束规范

> ⚠️ **必读**：本文件包含项目硬性约束，所有 agent 在工作前必须阅读并严格遵守。
> 
> 约束来源：开发工程师、测试工程师、产品经理、项目负责人四角色共同制定

---

## 1. 分支管理约束

### 1.1 基本规则
- **绝对不要**直接修改和合并到 `master` 分支
- 所有工作必须在 feature 分支进行
- 可以向 `master` 发起 PR，但**必须由用户确认后才能合并**
- 合并前必须通过所有测试和代码审查

### 1.2 分支工作规范
- **及时提交**：工作进度要及时 commit，避免大量未提交修改
- **及时推送**：commit 后及时 push 到远程分支，防止本地丢失
- **备份策略**（灵活判断）：
  - 重要改动、长时间工作：推送备份
  - 小规模修改、即将完成：可本地暂存
  - 不确定时：宁可多备份

### 1.3 分支清理规范（定期检查）
- **检查频率**：每周或每两周检查一次（不要太频繁）
- **检查对象**：长时间未使用的分支（如超过 2 周无活动）
- **处理方式**：
  - 不要直接删除分支
  - 评估分支是否还有保留价值
  - 如确定不再需要：**重命名分支**，添加 `nouse/` 前缀
  - 示例：`feature/old-thing` → `nouse/feature/old-thing`
- **豁免规则**：带有 `nouse/` 前缀的分支不再进行检查

## 2. 依赖版本管理

- 所有依赖版本的变化都要同步更新
- Python: 及时修改 `requirements.txt` 或 `pyproject.toml`
- Node.js: 及时更新 `package.json` 和 `package-lock.json`
- 新增依赖必须说明版本号和用途
- 提交前检查：`pip freeze` / `npm list` 确认版本一致性

## 3. 文档同步要求

  - [ ] `API.md` / `docs/api.md` - 接口文档
  - [ ] `ARCHITECTURE.md` - 架构设计文档

- 文档更新与代码修改应在同一 PR 中提交

## 4. 测试规范

### 4.1 初步实现阶段
- 允许简单测试接口响应情况
- 建议进行端到端测试验证基本流程

### 4.2 功能落地阶段（硬性要求）
- **必须**进行端到端测试
- **必须**模拟前后端交互
- **必须**模拟真实用户操作：前端点击按钮、填写表单等
- 重要测试结果**必须截图**（使用 Playwright 等工具真实截图，不能自己生成图）
- 截图需保存到 `test-screenshots/` 目录
- **截图必须先由用户检查确认**后才能标记测试通过
- 测试报告需向用户展示截图供确认

### 4.3 测试截图管理规范
- **Git 管理**：测试截图**不提交到 Git**，添加 `.gitignore` 排除
- **目录规范**：截图保存到 `test-screenshots/`（不在 `tests/screenshots/`）
- **清理策略**：每一轮测试前**必须清理**上一轮截图，避免混淆
- **保留期限**：截图仅用于当前轮次验证，不长期保留
- **用户确认**：截图必须先发给用户检查，确认无误后才能说测试通过

### 4.4 测试检查清单
每次提交前必须执行：

```bash
# 1. Python 语法检查
cd backend
python3 -m py_compile app/collaboration/*.py app/api/collaboration_routes.py

# 2. 后端启动测试
source .venv/bin/activate
python3 -c "from app.main import app; print('Import OK')"

# 3. 前端编译检查
cd ../frontend
npm run build

# 4. 端到端测试（功能落地时必须）
# - 启动前后端服务
# - 使用 Playwright 模拟用户操作
# - 截图保存到 test-screenshots/

# 5. Git 提交
git add -A
git commit -m "..."
git push origin feature/xxx
```

## 5. 多角色审视约束（新增）

### 5.1 开发工程师视角

#### 代码质量约束
- **架构设计**：新功能需考虑与现有架构的兼容性，避免技术债务
- **接口设计**：API 设计需遵循 RESTful 规范，保持向后兼容
- **错误处理**：所有异常必须有合理的错误处理和用户友好的提示
- **日志记录**：关键操作必须记录日志，便于问题排查
- **性能考虑**：数据库查询、循环操作需考虑性能影响
- **安全考虑**：用户输入必须验证，防止 SQL 注入、XSS 等攻击

#### 代码审查清单
- [ ] 代码是否符合项目编码规范
- [ ] 是否有适当的错误处理
- [ ] 是否有必要的日志记录
- [ ] 是否引入了不必要的依赖
- [ ] 是否考虑了边界情况
- [ ] 是否对性能有影响

### 5.2 测试工程师视角

#### 测试覆盖约束
- **单元测试**：核心功能必须有单元测试覆盖（覆盖率 > 70%）
- **集成测试**：模块间交互必须有集成测试
- **边界测试**：必须测试边界值和异常情况
- **性能测试**：性能敏感功能必须有性能基准测试
- **安全测试**：涉及用户数据的功能必须有安全测试

#### 测试通过标准
- 不能仅以"接口通"为通过标准
- 必须验证业务逻辑正确性
- 必须验证数据一致性
- 必须验证用户体验流程
- 必须验证错误处理流程

#### 测试文档
- 复杂测试场景需有测试用例文档
- 测试数据准备步骤需记录
- 已知问题和限制需明确标注

### 5.3 产品经理视角

#### 功能完整性约束
- **需求对齐**：实现必须与需求文档一致，如有偏差需确认
- **用户体验**：前端交互需符合用户习惯，操作流程需顺畅
- **功能边界**：明确功能边界，超出范围的功能需确认
- **优先级管理**：按优先级实现，P0 功能必须完整

#### 功能验收标准
- [ ] 功能是否符合用户需求
- [ ] 操作流程是否顺畅
- [ ] 错误提示是否友好
- [ ] 是否考虑了异常场景
- [ ] 是否需要用户培训

### 5.4 项目负责人视角

#### 项目管理约束
- **进度控制**：每周检查进度，延期风险及时上报
- **风险管理**：识别技术风险，制定缓解方案
- **资源管理**：评估资源需求（时间、人力、基础设施）
- **依赖管理**：外部依赖（如 E2B）需有备用方案

#### 决策确认清单
- [ ] 技术方案是否经过多角色评审
- [ ] 是否考虑了长期维护成本
- [ ] 是否有明确的验收标准
- [ ] 是否考虑了扩展性
- [ ] 是否有回滚方案

#### 约束执行检查
- **每日检查**：代码提交、测试执行、文档更新
- **每周检查**：进度回顾、风险识别、计划调整
- **每阶段检查**：功能验收、性能验证、安全审查

## 6. 沙箱功能专项约束

### 6.1 架构设计约束
- **适配器模式**：必须支持多种实现（E2B、Docker、Local）灵活切换
- **资源隔离**：沙箱执行必须有资源限制（CPU、内存、时间）
- **结果持久化**：执行结果必须保存，支持后续引用
- **会话集成**：执行结果必须融入会话历史

### 6.2 安全约束
- **代码审查**：沙箱相关代码必须经过安全审查
- **权限控制**：禁止的危险操作必须有明确限制
- **资源释放**：执行完成后资源必须及时释放
- **超时控制**：所有执行必须有超时限制

### 6.3 测试约束
- **多实现测试**：每个适配器实现必须独立测试
- **主备切换测试**：Composite 适配器必须测试故障转移
- **资源限制测试**：必须测试资源超限场景
- **并发测试**：必须测试多用户并发执行场景

---

## 7. 回滚策略

如果测试失败：
1. 记录错误信息
2. 使用 `git reset --soft HEAD~1` 回滚到上次提交
3. 修复问题
4. 重新测试
